# [Topology](./0.1.x.excalidraw)

# Environmental Assumption
Each node within the system is executed in an isolated Docker container. This configuration ensures that network behaviors and interactions are tested in a controlled and reproducible environment, free from external influences or conflicts between node processes.

# Methodology
The testing methodology outlined herein involves the orchestration and management of multiple `docker` containers to assess the interactions between them across two distinct, isolated networks.

## Network Configuration
Two networks, labeled Network `A` and Network `B`, are instantiated with specific configurations to facilitate IPv4 communication. These networks are isolated from each other and configured with no ingress traffic, ensuring that only explicitly routed connections between nodes are permitted.

## Container Initialization and Role Assignment
### Bootstrap
The `bootstrap` container is instantiated and connected to both Network `A` and Network `B`. Its role is to serve as the central coordination node for all other containers, facilitating initial connections and acting as a reference point for the other nodes.

### Relay
A set of `relay` containers are launched, with a 25% chance that any given relay will be classified as a `malicious_relay`. These containers are connected to both networks and are configured to establish communication with the bootstrap container. The intentional inclusion of `malicious_relay`s allows for the simulation of adversarial conditions in the network.

### Server
A `server` container is instantiated and connected to Network B, where it listens for incoming gRPC requests from the `client` container. The `server` is responsible for responding to `client` queries and simulating the core service functionality within the network.

### Client
The `client` container is launched and connected to Network `A`. It periodically issues gRPC requests to the `server` via the `bootstrap` container, thereby simulating client-server communication within the network.

## gRPC Communication
### Client-Server Interaction
The `client` container initiates periodic dial requests to the `server` container via gRPC, simulating a realistic pattern of communication and data exchange between the two nodes.

### Role of the Bootstrap Container
The `bootstrap` container serves a vital role in managing the initial connection setup for both the `client` and `server`. It functions as a rendezvous point, ensuring that the `client` and `server` can establish a secure communication path across the network.

## Logging and Proof Collection
### Log Generation
Detailed logs are collected during the test execution and stored in a predefined directory (`log_dir`). These logs capture important network events, including node startup, connection attempts, and data exchanges between containers.

### Proof Analysis
The collected logs are subsequently analyzed to determine:
- Successful startup and initialization of containers.
- Service cohesion and successful communication between `client` and `server`.
- Network stability and the persistence of connectivity across the system.

## Cleanup Procedure
Upon the completion of the test, all created networks are released, and any instantiated containers are cleaned up. The logs generated during the test are preserved for further analysis and reporting.

> Note: In the event that the test fails before cleanup, it is necessary to manually clean up any unused networks to ensure that the system environment remains clean for future tests.

# [Success](../../../../doc/MILESTONE.md)

# Observation
## Service Cohesion
The `bootstrap` container was effective in its role as the central coordination point for all other containers. The `client` successfully initiated dial requests to the `server` via the `bootstrap` container, demonstrating the proper setup of inter-container communication. Additionally, the random selection of `malicious_relay`s worked as expected, introducing adversarial conditions that did not interfere with the overall communication between the `client` and `server`.

## Network Cohesion
The network topology was verified to be cohesive. All nodes in the simulation were interconnected, with no fragmentation in the network. Every node maintained a valid connection to at least one other node, ensuring complete network connectivity.

## Churn and Stability
Over time, the network demonstrated a tendency to stabilize, with churn (the frequency of node reconnections) decreasing as the system reached a steady state. Each node was designed to maintain a minimum peer count. In scenarios where the number of peers fell below this threshold, nodes would continually attempt to find additional peers, leading to temporary churn. Nevertheless, even under high churn conditions—where nodes attempted to reconnect every millisecond—the system maintained acceptable efficiency and stability.

## gRPC Control Plane
The `client` container was able to successfully establish communication with the `server` through both the `bootstrap` and `relay` containers. Importantly, the presence of malicious `relay` containers did not prevent successful communication, indicating a level of resilience in the system's design. This suggests that the architecture is capable of handling adversarial conditions without significant disruption to core functionality.

# Reproduce

Ensure the following tools are installed and working:

- `rustup`
- `cargo`
- `protoc`
- `docker`

Clone the repository:

```
git clone https://github.com/ravendevteam/alternet
```

Execute the test using the following command at the workspace root:

```
cargo test --features=end_to_end
```

This command will:
- Build a fresh Docker image reflecting the current state of the nodes.
- Automatically run the test, establishing connections between the containers and performing network interactions as described in the methodology.
- Analyze the generated logs to verify the network behavior and architectural feasibility.

## Log Analysis

The test will generate logs that are stored in the `log_dir` directory. 

These logs contain detailed reports on:
- The successful startup and initialization of all containers.
- The successful interaction between the `client` and `server`.
- The stability of the network, including connectivity persistence over time.

These logs provide critical proof of the system's ability to function under the specified conditions, supporting the validation of the network's design and its resistance to adversarial actions.
